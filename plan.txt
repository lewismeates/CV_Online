2. Technology Stack Decision
2.1 Recommended Engine: Godot 4 (GDScript)
After evaluating the constraints — macOS desktop target, portfolio goal, solo/small team, Python/Java/JS background, 3D requirement, 5v5 gameplay — Godot 4 with GDScript is the clear recommendation. The rationale is as follows:

Why Godot 4 over the alternatives
Engine
Pros
Cons / Why Not
Godot 4
Free, open-source, native macOS .app export, excellent 3D pipeline, GDScript similar to Python, huge community, lightweight, fast iteration

Massive asset store, great 3D, well documented

2.2 Language: GDScript
GDScript is Godot's native scripting language. It is syntactically nearly identical to Python — indentation-based, dynamically typed, and readable. For statically typed, performance-critical systems (physics, AI), GDScript supports optional type annotations. This makes the transition from Python trivial.
	•	GDScript is to Godot what JavaScript is to the browser — it is the native language and using it avoids all C# integration overhead.
	•	All Godot 4 documentation, tutorials, and community answers are GDScript-first.
	•	Optional static typing (var x: float = 0.0) gives performance benefits in hot paths.

3. Architecture: ECS-Influenced MVC for Game Dev
3.1 Why Not Pure MVC?
Classical Model-View-Controller is well-suited to UI applications but breaks down in real-time games for a key reason: the game loop runs 60 times per second, and routing every frame update through a controller creates latency and boilerplate. The game development community has converged on a hybrid approach.
For this project the recommended pattern is: Godot's Node hierarchy as a natural scene graph, with explicit separation of concerns using a service-locator pattern via Autoloads and a data layer via Resource objects. This gives the readability benefits of MVC with the performance and simplicity of a game-native architecture.

3.2 Architecture Layers
Autoloads (Singletons)
GameManager, PossessionManager, ScoreManager, InputManager. These are the 'Controllers'. They own game-wide state and emit signals. No scene depends on another scene directly — they all talk through Autoloads.
Scene Nodes (Views)
Court, PlayerMesh, HUD, ShotMeterUI, PassUI. These are the 'Views'. They listen to signals from Autoloads and update their visual state. They own no game logic.
Resource Objects (Models)
PlayerData, TeamData, GameConfig. These are the 'Models'. Pure data containers, no Node overhead, serialisable to disk, easily unit-tested.
Component Scripts
DribbleController, ShootController, AIBrain, DefenseAgent. Attached to player nodes as child scripts. Stateless where possible — they read from the Model and emit to the Controller.
Signal Bus
All cross-system communication goes through Godot signals. No direct node.get_node('/root/X') chains in gameplay code. This is enforced by the linter rule set.

4. Codebase Development Rules
These rules are non-negotiable. Every story must be implemented in compliance with all of them. They are designed to be enforced automatically where possible (linter, CI) and reviewed manually where not.

4.1 Testing Rules
	•	Every script that contains logic (not just visual output) must have a corresponding test file in tests/unit/.
	•	Test files are named test_<script_name>.gd and use the GUT framework.
	•	Minimum coverage requirement: all public functions, all state transitions, all edge cases listed in acceptance criteria.
	•	CI will run the full GUT suite headlessly on every push. A failing test blocks the merge.
	•	Performance tests: any system that runs in _physics_process() must have a benchmark test asserting it completes in < 2ms per frame.
	•	No story is 'done' until its tests pass on CI. Done = coded + tested + merged.

4.2 Naming Conventions
Files / Scenes
snake_case.gd, snake_case.tscn, snake_case.glb
Classes
PascalCase — class_name PlayerAgent
Variables
snake_case — var shot_quality: float
Constants
SCREAMING_SNAKE — const MAX_SPEED: float = 8.0
Signals
past_tense_snake — signal shot_released(quality: float)
Enums
PascalCase type, SCREAMING_SNAKE values — enum PlayerState { IDLE, DRIBBLING }
Functions
snake_case, verb-led — func calculate_shot_arc() -> Vector3
Test functions
test_ prefix — func test_shot_quality_returns_zero_on_miss()

4.3 Code Style Rules
	•	Maximum function length: 30 lines. If longer, extract a helper function.
	•	Maximum file length: 200 lines. If longer, split into component scripts.
	•	No magic numbers. Every constant must be named and placed in a Constants.gd autoload or local const block.
	•	Use static typing on all function signatures and class variables. Dynamic typing is only permitted in genuinely polymorphic contexts.
	•	All signals must be documented with a one-line comment above the emit() call explaining what triggered it.
	•	No direct node path access (/root/GameManager). Use autoload references or dependency injection via exported variables.
	•	Every public function must have a one-line docstring comment above it.

4.4 Git Workflow
	•	Branch naming: feature/US-XXX-short-description, bugfix/US-XXX-description, chore/description
	•	Commit messages: US-XXX: <imperative verb> <what changed>  e.g.  US-043: add shot meter fill animation
	•	One story = one branch = one PR. No batching multiple stories in one PR.
	•	PRs require: green CI (lint + tests), self-reviewed diff, and a one-line description of what was changed and why.
	•	main branch is always deployable. Never commit broken code to main.
	•	Squash merge only — keeps history clean and traceable to story IDs.

4.5 Architecture Rules
	•	Game logic lives in Autoloads and Component scripts. Scene nodes are dumb — they render state, they do not own it.
	•	No scene communicates directly with another scene. All cross-scene communication uses signals routed through Autoloads.
	•	Resource files (.tres) are the single source of truth for all tunable values (speeds, distances, probabilities). No hardcoding gameplay values in scripts.
	•	The ball is always owned by exactly one entity: a PlayerAgent, CPUAgent, or the physics world. The PossessionManager enforces this as an invariant.
	•	Every Autoload must implement a reset() method that returns it to a clean state, to support the Restart flow without scene reloads.

4.6 Performance Rules
	•	Target: 60fps on Apple Silicon M1 or equivalent. All decisions are made with this constraint in mind.
	•	No per-frame heap allocations in _process() or _physics_process(). Pre-allocate arrays and objects in _ready().
	•	CPU AI decisions run at 10Hz (every 6 physics frames), not 60Hz. Use a timer-gated update pattern.
	•	Particle systems use pooling (US-096). No instantiate() calls in hot paths.
	•	3D player meshes must stay under 500 triangles for primary LOD, 150 triangles for distant LOD.

5. Sprint Plan (Recommended Execution Order)
Stories are grouped into sprints by dependency order. Each sprint builds on the last and produces a testable, runnable increment. An AI agent should execute one sprint at a time and run tests before proceeding.

Sprint 1 — Foundation
US-001 through US-007. Project setup, repo, CI, test framework. Nothing renders yet. Goal: green CI pipeline.
Sprint 2 — See the Court
US-008 through US-015, US-026, US-087. Court renders, hoop is in scene, ball exists, camera works. Goal: something to look at.
Sprint 3 — A Player Exists
US-016 through US-025, US-031 through US-033. One low-poly player on court, can move, has idle and run animations. Goal: first human control.
Sprint 4 — Ball in Hand
US-027 through US-029, US-037 through US-040. Player can pick up the ball, dribble while moving. Goal: core possession loop.
Sprint 5 — Shoot the Ball
US-043 through US-050, US-079, US-080. Shot meter UI, shooting arc physics, rim collision. Goal: can score points.
Sprint 6 — Pass the Ball
US-051 through US-056, US-082. Pass UI, chest pass, auto-catch. Goal: ball movement between players.
Sprint 7 — CPU Enters
US-057 through US-062. CPU offense: NavAgent, drive, space floor, shoot. Goal: CPU can score.
Sprint 8 — CPU Defends
US-063 through US-069. Man-to-man defense, closeout, steal, contest shot. Goal: playable 1-half game.
Sprint 9 — Rules & State
US-070 through US-078. GameManager, clocks, possession changes, inbound, game over. Goal: full game loop.
Sprint 10 — Polish
US-083 through US-086, US-088 through US-094, US-095 through US-100. HUD polish, audio, performance, export. Goal: shippable portfolio build.

7. Definition of Done
A user story is considered Done only when ALL of the following criteria are met:

	•	The feature behaves as described in the 'I want to...' and 'So that...' fields when tested manually in the Godot editor.
	•	All acceptance criteria listed in the story have been verified.
	•	At least one automated GUT test covers the core logic path of the story.
	•	gdlint reports zero warnings on all files touched by the story.
	•	The feature runs at >= 60fps on a test scene with all 10 players active (verified by profiler).
	•	The branch has been merged to main via a squash-merge PR with green CI.
	•	If the story changes a public API (signal, function signature, Resource schema), CHANGELOG.md has been updated.

8. Key Risks & Mitigations
3D character rigging complexity
Use Mixamo's free auto-rigger for the base skeleton. Retopologise in Blender to hit the 500-tri budget. This is the highest-effort asset task — do it in Sprint 3 and block nothing else on it.
CPU AI pathfinding on a small court
Godot's NavigationServer3D can struggle with dense, small meshes. Bake a simplified NavMesh with generous agent radius. AI runs at 10Hz to mask any pathfinding latency.
Ball physics feel
Basketball physics are notoriously hard to tune. Reserve 1 full session for physics parameter tuning after Sprint 4. Key values (bounciness, friction, dribble impulse) all live in a PhysicsConfig.tres resource for easy iteration.
macOS notarisation
Apple's notarisation process requires a paid Developer account ($99/yr). If unavailable, ship with a README note instructing users to right-click > Open to bypass Gatekeeper. Plan for this in Sprint 10.
Animation state machine complexity
With idle, run, dribble, shoot, pass, and defend, the AnimationTree can become a spaghetti graph. Design the blend tree in Sprint 3 before any animations are wired, and document it in a diagram committed to docs/.

9. Agent Execution Instructions
When handing this document to a coding agent, the following instructions apply:
	•	Execute one sprint at a time. Do not begin Sprint N+1 until all Critical and High stories in Sprint N have green tests.
	•	After each story, run gdlint on all modified files. Do not proceed if lint fails.
	•	After each sprint, run the full GUT test suite. Paste the result into a SPRINT_LOG.md file in the repo root.
	•	If a story's acceptance criteria are ambiguous, implement the most conservative interpretation and add a TODO comment with the ambiguity for human review.
	•	All .tres resource files for tunable values should be created in Sprint 1 (as empty shells) and populated as the relevant stories are implemented.
	•	Do not skip Low priority stories permanently — flag them as deferred in a DEFERRED.md file if time-constrained.
	•	Every commit message must reference the story ID (e.g. US-043: add shot meter fill animation). This is enforced by a commit-msg git hook set up in US-001.
